# Contributing to Ориентир

Спасибо за интерес к проекту! Этот документ описывает, как можно помочь.

## Что мы принимаем

- **Баг-репорты** — найдена уязвимость, обход защиты, false positive в расширении.
- **Идеи и предложения** — новый уровень защиты, новый паттерн манипуляции, улучшение UX.
- **Кейсы** — личные истории взаимодействия с ИИ (анонимно опубликуем в блоге).
- **Контент** — новые статьи, улучшение существующих, перевод (но см. ниже).
- **Код** — bugfixes, новые функции, оптимизация.

## Чего мы НЕ принимаем

- **«Хороший ИИ-агент».** Это принципиальная позиция проекта — мы строим фильтр, не агента. PR с ИИ-агентом будет отклонён.
- **Трекеры и аналитика.** Никаких GA, Метрики, Sentry (в продуктивном коде). Локальная статистика — ок.
- **Закрытый код.** Всё, что попадает в репозиторий, — под MIT.
- **Изменения манифеста без обсуждения.** Если меняете философию — откройте issue сначала.

## Как сообщить о баге

1. Проверьте [существующие issues](https://github.com/orientirtech-afk/orientir/issues) — возможно, уже сообщили.
2. Откройте новый issue по шаблону `Bug report`.
3. Укажите: версию расширения (chrome://extensions/), браузер + версию, ОС, AI-сайт, где воспроизвелось, конкретные шаги, ожидаемое и фактическое поведение.
4. Если есть — прикрепите скриншот или запись экрана.

## Как предложить новую функцию

1. Откройте issue по шаблону `Feature request`.
2. Опишите: что не так сейчас, что вы предлагаете, почему это в духе проекта.
3. Если есть — ссылка на исследование, статью, прецедент.

## Как предложить новый паттерн манипуляции

1. Откройте issue по шаблону `Pattern submission`.
2. Опишите: название паттерна, пример (реальный текст — анонимизированный), какая контрмера работает.
3. Мы добавим паттерн в [Справочник](https://orenpro.pro/pages/reference.html) и, возможно, напишем разбор в блоге.

## Как добавить статью в блог

1. Создайте файл `pages/blog/<slug>.html` по образцу существующих статей. Скопируйте структуру header/footer/nav из любой статьи.
2. Добавьте `<link rel="stylesheet" crossorigin href="/assets/css/print.css" media="print">` в `<head>` для print-friendly версии.
3. Обновите `pages/blog/index.html` — добавьте карточку в `.blog-grid`.
4. Обновите `feed.xml` — добавьте `<item>` с метаданными.
5. Обновите `sitemap.xml` — добавьте `<url>` с `lastmod`.
6. Отправьте PR.

**Требования к контенту:**
- Соответствие манифесту (внешний фильтр, не ИИ-агент).
- Минимум 150 слов на раздел, без однострочных абзацев.
- Проверяемые факты (ссылки на источники).
- Уважение к читателю — без поучения, без запугивания.

## Как добавить кейс

1. Создайте `pages/blog/case-<slug>.html` с `data-category="case"`.
2. Кейс — личная история. Не «разбор чужого текста», а «что я заметил у себя».
3. Анонимность: если кейс от читателя — не указывайте имени, если он сам не попросил.
4. Структура: контекст → симптомы → что сделал → вывод.
5. Обновите блог-индекс, `feed.xml`, `sitemap.xml`.

## Как внести код в расширение

1. Fork → branch → commit → PR.
2. Один PR — одна функция. Не смешивайте refactor и feature.
3. Перед PR:
   - `node --check` на всех изменённых JS-файлах.
   - Проверьте JSON-валидность `manifest.json` и `rules/telemetry_rules.json`.
   - Если меняете `manifest.json` — обновите `CHANGELOG.md`.
4. Описание PR: что делает, зачем, как тестировали.

### Архитектура расширения

```
extension/
├── manifest.json              # Manifest V3, два content_scripts
├── background.js              # Service worker, DNR onRuleMatchedInfo, alarms
├── content-bridge.js          # ISOLATED world — мост MAIN ↔ background
├── content.js                 # MAIN world — перехват API
├── rules/telemetry_rules.json # DNR-правила
├── popup/                     # UI popup (storage.onChanged)
└── options/                   # Настройки (тогглы + whitelist + export/import)
```

**Важно:** MAIN world не имеет доступа к `chrome.*` API. Все сообщения — через `window.postMessage` в bridge, bridge — через `chrome.runtime.sendMessage` в background.

## Стиль кода

- JavaScript: ES-модули, без транспиляции, без TypeScript (пока).
- 2 пробела для отступов.
- Комментарии на русском (как в существующем коде).
- Имена функций и переменных — на английском.

## Стиль текстов

- Русский язык, без кальки с английского.
- Без канцелярита («в данном случае», «осуществляется»).
- Без эмодзи в основном тексте (только в навигации/иконках).
- Цитаты — в кавычках-ёлочках.
- Длинные предложения разбивайте на два.

## Лицензия

Внося вклад, вы соглашаетесь, что ваш код будет под MIT, а тексты — под CC BY 4.0.
